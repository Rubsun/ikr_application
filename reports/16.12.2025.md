# Отчет по Code Review пакетов проекта

## Требования из readme.md

### Архитектура:
- **ui** - слой логики отображения данных
- **domain** - слой рабочей логики приложения
- **data** - слой для получения "сырых" данных или dto от внешнего API или из DB
- **Каждый слой знает только про тот слой, который находится ниже**

### Требования к проекту:
1. Создать свой "пакет" и добавить туда три подпакета: **data**, **domain**, **ui**
2. В своем пакете, в data, создать **Repository** который пока будет генерить фэйковые данные
3. В своем пакете, в domain, создать несколько **UseCase** которые должны будут из Repository брать данные и преобразовывать к обычным данным
4. В своем пакете, в ui, создать **Fragment**:
   - Добавить на него **ViewModel**, который будет получать данные из UseCase-ов
   - Добавить в него верстку (можно Compose, можно xml), которая отображает данные из ViewModel
   - Настроить маппинг данных от ViewModel к верстке
5. При создании верстки - учесть темы и стили
6. Убрать использование LiveData! Не использовать LiveData!
7. Получать данные на фрагмент при помощи Flow!
8. Получать данные на фрагмент в виде State-объекта, из которого данные должны будут мапиться в отображение
9. Добавить в верстку элемент, который позволит изменять состояние данных. И сделать на фрагменте его обработку:
    - Как пример - поисковая строка, которая будет аффектить на то, как фильтруются данные
    - Как пример - сделать блок, через которые можно будет добавить данные в репозиторий, а ViewModel вернет эти данные в список
    - После изменения данных - команда отправляется в ViewModel и через UseCase они или добавляются в репозиторий, либо используются как фильтр
10. Добавить использование внешней библиотеки
    - Как пример - библиотека, которая грузит данные
    - Как пример - библиотека, которая помогает отобразить картинки
    - Как пример - библиотека, которая рисует графики
    - Все библиотеки должны добавляться при помощи гредл-каталога!
    - Если добавляете использование какого-то внешнего API с tokenom - нельзя его токен сохранять в коде, надо добавить инструкцию в этот readme, как получить токен (что бы я мог проверить:)
11. Если делаются какие-то тяжелые операции - то при помощи suspend и Dispatcher переводить эту работу в IO или Default потоки
12. Изменения в main добавлть через PullRequest!

### Штрафы
1. Нарушение каждого пункта из требований - 2 балла (за каждый пункт)
2. Использование чужиж класса или верстки - 2 балла (за каждый юнит)
3. Создание лишних файлов - 1 балл (за каждый юнит)
4. Копипаста файла (полное совпадение структуры файла) - 1 балл (за каждый юнит)
